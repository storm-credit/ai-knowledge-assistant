# 14. 적용형 학습 노트 — 스펙 (v1)

> "하네스 엔지니어링 같은 새 개념이 뜨면, 이게 뭔지 + 어떻게 배울지 + **내 스킬로 어떻게 써먹을지**를
> 시스템이 정리해준다." 학습 카드(12)가 영상 1개 요약이라면, 이건 **개념 1개를 깊게 파는 노트**다.
> PLAYBOOK 7절 **AI 전문가 eval 트랙을 처음 적용하는 기능.**

- 작성: 2026-07-02 · 전제: 작업계획서(13) Batch 1의 쿼터 예산(카드 A) 가동 후 착수

## 미결정 5건 → 확정 (전문가 패널 소견 반영)

| # | 결정사항 | 확정 |
|---|---|---|
| 1 | **내 스킬 프로필** | `notes/me.md` 자유 텍스트 2~3줄 (예: "파이썬·Claude Code로 개인 도구 만듦, PM 관점 선호"). **비어 있어도 작동**(그땐 일반적 적용법) — 채울수록 맞춤도↑. 프롬프트에 원문 그대로 주입, 사람이 직접 관리(LLM이 과거 노트에서 추론하지 않음 — 버전 관리 가능해야). |
| 2 | **발동·신규성 기준** | v1 = **온디맨드 수동 지정만**: `python -m collector learn "하네스 엔지니어링"`. 자동 후보 추천(피드 반복 등장 개념)은 쿼터를 쓰므로 v2로. |
| 3 | **근거 규칙 (핵심 설계)** | 섹션별로 다르게 — **grounded/generative 분리**: `이게 뭔가`·`관련`은 피드 맥락+모델 일반지식 허용하되 **불확실·최신 정보는 표시**, `어떻게 배우나`는 일반지식 허용, `내 적용법`은 me.md 기반 **추론 허용 + 추측임을 명시**. (학습 카드의 "지어내지 마라" 전면 적용은 이 기능에선 오히려 해로움) |
| 4 | **eval 방식** | 골든 개념 2건(예: "프롬프트 엔지니어링", "MCP") 고정 → `pytest -m llm`(환경변수 게이트, cron 미실행)으로 실 API 후 **코드 검증**: 4섹션 존재, 내 적용법에 me.md 키워드 반영, 추측 표시 존재, 판촉어 0. 출력 스냅샷 저장해 프롬프트 수정 전후 diff 리뷰. |
| 5 | **쿼터 할당·출력 위치** | 노트당 **1콜**, 실행 시점 일일 예산 잔여 ≥2콜일 때만 (본업인 아침 수집 우선). 출력: `notes/learn/<개념>.md` **독립 페이지**(위키 병합 안 함 — 헤딩 자유) + 웹 뷰어에 `/learn` 목록·상세 노출. |

## 노트 구조 (4섹션)

```markdown
# <개념명>
> 생성일 · 요청 계기(어느 피드 항목에서 봤나, 있으면)

## 이게 뭔가          ← 개념 정리 (피드 맥락 + 일반지식, 불확실 표시)
## 어떻게 배우나      ← 학습 순서·경로 (뭘 먼저, 뭘 실습)
## 내 적용법          ← me.md 기반 "너는 X 하니까 이렇게" (추측 표시)
## 관련               ← 내 위키의 기존 개념 [[링크]]
```

## 구현 스케치 (Batch 1 완료 후 작업 카드로)

- `collector/learn.py` — 개념명 + me.md + 최근 피드 관련 항목(제목·요약 로컬 검색) → LEARN_PROMPT 1콜 → notes/learn/ 저장. LLM 호출은 `collector/llm.py`(카드 A) 경유, 예산 확인.
- `collector/__main__.py` — `learn "<개념>"` 서브커맨드.
- `web/` — /learn 목록·상세 (기존 render 재사용).
- `tests/` — FakeClient 구조 테스트 + `-m llm` eval 2건.
