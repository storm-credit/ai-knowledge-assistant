# AI 프로젝트 진행 플레이북 (재사용 템플릿)

> **목적:** 어떤 새 프로젝트든 이 순서대로 진행하면 길을 잃지 않는다.
> 새 프로젝트 시작할 때 이 파일을 복사해서 `docs/`에 단계별 문서를 만들어 나가면 된다.
> (이 플레이북은 "개인 AI 지식 비서" 프로젝트를 실제로 진행하며 정리한 것)

## 0. 시작 전 3대 원칙

1. **WHAT과 HOW를 분리한다.** 무엇을 만들까(WHAT)는 내가(PM), 어떻게 만들까(HOW)는 엔지니어(에이전트)에게 위임. 머리 아픈 대부분의 기술 단어는 HOW다 — 지금 다 정하려 하지 말 것.
2. **한 번에 다 만들지 않는다.** 가장 작은 "쓸모 있는 한 조각"을 먼저 완성하고 얹는다.
3. **단계마다 문서를 남긴다.** (이 플레이북이 그 증거)

## 1. 단계 ↔ 산출 문서 ↔ 핵심 질문

| 단계 | 산출 문서 | 핵심 질문 | WHAT/HOW |
|---|---|---|---|
| 1. 기획(브레인스토밍) | (대화 + 메모) | 누구를 위해? 진짜 목적? 가장 작은 1단계는? | WHAT |
| 2. 컨셉/아키텍처 | `01-concept-and-architecture.md` | 데이터가 어떻게 흐르나? 어떤 부품? 로드맵? | WHAT |
| 3. 요구사항 정의서 | `02-requirements-spec.md` | 구체적으로 뭘 해야 하나? 성공 기준? | WHAT |
| 4. 구현 계획 | `03-implementation-plan.md` | 어떤 스킬/MCP/훅/에이전트로? 작업 분해? | HOW(큰그림) |
| 5. 구현/튜닝 | (코드 + 산출물) | 프롬프트/컨텍스트/에이전틱 튜닝 | HOW(디테일) |

## 2. 결정 타이밍 표 — "이 개념은 언제 정하나"

| 개념 | 정하는 단계 | 메모 |
|---|---|---|
| 하네스(에이전트 실행 틀) | 0/시작 시 | 예: Hermes, Claude Code |
| 스킬(Skills) | 2~4 | 능력 모듈. 아키텍처에서 일부 매핑, 구현계획에서 확정 |
| MCP(외부 연동) | 4 (구현계획) | 기본 도구로 되면 안 붙임 |
| 훅(Hooks, 자동 트리거) | 4 (구현계획) | 필요시만 |
| 서브에이전트 역할(아키텍트/코더/리뷰) | 4 (구현계획) | 빌드 팀 = 에이전트가 대신 |
| 벡터DB | 검토 후 필요시 | 데이터 적으면 불필요 |
| LangChain 등 프레임워크 | 4 | 하네스가 이미 하면 보통 불필요 |
| 프롬프트/컨텍스트/에이전틱 엔지니어링 | 5 (구현) | 만들면서 튜닝 |

## 3. 두 종류의 "에이전트" (헷갈리지 말 것)

- **제품 안에서 돌아가는 에이전트(런타임):** 제품의 부품. 예: 수집/요약/Q&A 에이전트. → **2~3단계에서 설계.**
- **만드는 팀 에이전트(빌드타임):** 아키텍트·DBA·UI·코더·리뷰어 역할. 전통 PM의 "사람 팀"을 에이전트가 대신. → **4단계 구현계획에서 배정.**

## 4. PM 검토 지점 (어디서 확인하나)

- **요구사항 검토** → `02-requirements-spec.md`의 `[확정필요]` 항목.
- **기술 구성요소 검토(MCP/스킬/훅/에이전트)** → `03-implementation-plan.md`의 "구성요소 결정표".
- **최종 작업 분해 검토** → 구현계획의 Task Breakdown.

## 5. 새 프로젝트에 재사용하는 법

1. 새 폴더 `C:\ProjectS\<프로젝트명>\docs\` 생성.
2. 이 `PLAYBOOK.md` 복사.
3. `00-overview.md`(진행 보드) → `01`(아키텍처) → `02`(요구사항) → `03`(구현계획) 순서로 채운다.
4. 각 단계 끝에 진행 보드(00) 상태 갱신.

## 6. 실행 루프 — 서브에이전트 주도 개발 (HOW를 굴리는 법)

구현계획(4단계)에서 작업을 **독립된 조각(작업 카드)**으로 쪼갠 뒤, 한 조각씩 이 루프로 돈다:

1. **작업 카드 1개 선택** — 입력·출력·완료조건이 명확한 최소 단위 (예: "config.Source에 learning 플래그 추가 + 테스트").
2. **코더 서브에이전트에 위임** — "이 작업만, TDD로(테스트 먼저)". 컨텍스트는 그 작업에 필요한 것만 넘긴다.
3. **리뷰 서브에이전트로 검증** — *다른* 에이전트가 "요구 충족? 버그? 회귀?" 점검. (자기가 짠 코드는 스스로 후하게 보므로 리뷰는 반드시 다른 에이전트.)
4. **PM(나) 게이트 확인** — 테스트 초록불 + 리뷰 통과 → OK. 아니면 되돌림.
5. **통합·커밋 → 다음 카드.**

- **원칙:** 한 번에 한 작업. 작게 쪼갤수록 에이전트가 안 헤맨다.
- **병렬:** 서로 의존 없는 카드는 여러 에이전트를 동시에 띄운다. 의존 있으면 순차.
- **리서치도 카드다:** "채널ID 찾아 검증" 같은 조사도 리서치 에이전트에 위임 (이 프로젝트 실제 사례).

## 7. AI 전문가는 따로 기획하나? — "별도 기획"이 아니라 "붙는 트랙"

PM은 **무엇·왜·성공기준·우선순위**를 잡는다. 그런데 LLM 기능은 결과가 **불확실**하다(같은 프롬프트도 결과가 흔들림). 그래서 AI 전문가는 *별도 기획서*를 쓰는 게 아니라, **구현계획 안에 "품질 트랙"을 병렬로 얹는다**:

1. **eval 먼저 — 성공기준을 측정가능하게.** "좋은 학습 노트"가 뭔지 체크리스트·골든샘플·LLM 심사로 정의. 이게 없으면 튜닝이 감(感)이 된다.
2. **작은 실험(스파이크).** 코드 짜기 전에 프롬프트 2~3안을 샘플 몇 개에 돌려 비교.
3. **컨텍스트/데이터 설계.** 모델에 뭘 넣을지 — 프로필, few-shot 예시, RAG.
4. **회귀 방지.** eval을 테스트처럼 상시 돌려, 프롬프트를 바꿔도 품질이 안 떨어지게.

> **정리:** PM이 "성공기준"을 말하면, AI 전문가가 그걸 **숫자·샘플로 바꾸고 실험으로 HOW를 좁힌다.** 같은 계획서 안의 병렬 레인이지, 따로 노는 기획이 아니다.
> (지금까지 우리는 *구조* 테스트(pytest)만 있고 *품질* eval은 없다 → 학습 노트 기능에서 이 트랙을 처음 붙일 지점.)
